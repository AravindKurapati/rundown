# Rundown: solution notes

## The problem I chose to solve well

The brief asks for a personal podcast generator. The tempting version is a feature buffet: five news sources, three voice modes, a scheduler with a UI clock. I built the other version: one genuinely good daily briefing, produced by a pipeline where every cost, every failure, and every trade-off is visible. The grading signal was explicit about caring whether I understand the decisions behind the solution, so I optimized for decisions being legible, not for feature count.

The result is deliberately narrow at the paid boundary: one structured LLM call selects the stories and writes the whole script, and one ElevenLabs call per segment renders it. That per-segment split is a revision of the original design, which made exactly one TTS call per episode, and I changed it on listener feedback rather than on taste: the single-call read was flat, every story at the same pitch regardless of how much the listener cared. Because ElevenLabs bills per character, splitting the narration is cost-neutral; what it buys is a script whose per-segment energy actually reaches the voice, with each call conditioned on the neighboring script text so the prosody carries across calls and it stays one host. At the default rates an episode is still well under a dollar, and the dashboard now reports the provider's own billed character counts per call rather than my estimate of them.

## Architecture in one breath

Gather pulls Google News RSS per interest, a deterministic dedupe collapses the overlap, one LLM call turns the candidates into a segmented script, one TTS call per segment turns the script into stitched audio that is assembled and encoded locally, and the generator persists the MP3 plus per-stage timing and cost to SQLite. Every seam in that sentence exists for a reason. The news source is a protocol, so swapping RSS for a paid news API is a drop-in change, not surgery. The LLM and TTS clients are protocols with fake implementations, so all development, tests, and CI run offline and free; real spend is reserved for moments a human is listening. And the script is a list of segments, each tagged with a speaker and an energy, which makes the whole pipeline host-agnostic and delivery-aware at once.

## What ships, and what is designed but deferred

Single-host is the shipped, working product: one voice, expressive per-segment narration. Two-host is where I drew the line deliberately. The segment contract already carries a `speaker` on every segment, the prompt can write an alternating `host_a` and `host_b` script, and there is a sample two-host script in the repo (`docs/sample-episode-two-host.md`) to prove the writing works. The segmented narration engine has since closed most of the mechanical gap (per-segment synthesis, request stitching, local PCM assembly and encoding). What remains unbuilt is what makes two-host actually sound like two hosts: routing segments to per-speaker voices and loudness-matching two different voices so the handoffs do not jump. The text conditioning also assumes one continuous voice, so cross-speaker continuity is an open question I would want to prototype, not promise. So I shipped the honest single-host path and removed the "two hosts" control from the UI rather than leave a button that produces a two-person dialogue read by a single voice.

Scheduling got the same treatment. Instead of embedding a scheduler in the web process, and inheriting its restart, duplicate-run, and timezone problems, the schedule is persisted preference data with a computed next-run readout, and generation is a headless CLI that cron or Task Scheduler invokes through the same code path as the API. The product concept ships; the operational liability does not.

## Decisions I am glad I made

Merging selection and writing into one call is the one I would defend hardest. Splitting them sounds cleaner, but the editor and the writer being the same model is why the episodes have pacing: the thing choosing story order is the thing writing the transitions. It also halves the calls on the critical path.

Expressive segmented narration is the decision I made last and the one that best shows how I want to work: change the design when evidence says to, and say so. A listener told me the read was flat and the topics blurred together. The first response was cheap and global (real voice settings on the call, a prompt that writes speech rather than prose); the second was structural: the script now tags every segment with an energy that tracks the listener's interests, each segment renders as its own TTS call with the voice settings bent by that energy, and the audio is assembled locally with a beat of air and a quiet generated ding between topics. I evaluated Eleven v3's inline audio tags as the one-call alternative and rejected it on constraints, not vibes: a 5,000-character cap that cannot fit a ten-minute episode, and no continuity mechanism to chunk around it.

Continuity across the calls taught me the difference between reading the docs and running the code. I built this on ElevenLabs request stitching (`previous_request_ids`) after checking it was supported on our model, and the very first real generation failed with a 400: the provided account has high privacy mode enabled, and request-id stitching is rejected outright on such accounts because it references stored request history. Two things came out of that run. First, the failure accounting worked in production on its first try: the run died on segment two and billed exactly the 151 characters the intro had already spent. Second, the fix was to condition on the surrounding script text instead (`previous_text`/`next_text`), which is in-request context with no history dependency, so it works under any privacy setting. The seams are smooth rather than seamless, and the energy deltas are deliberately modest so adjacent segments sound like the same host getting more animated. And because assembly happens in PCM on this machine, episode duration went from a words-per-minute estimate to an exact measurement, and the cost ledger went from my estimate of characters to the provider's own `character-cost` header per call.

The fakes earned their keep within days. Because the fake TTS records its calls and returns real PCM bytes, tests can assert the load-bearing properties of the whole design, one synthesis call per segment with each call conditioned on the right neighbors, without a network or a dollar involved. The same fakes back a repro test for a real bug: OpenAI's JSON-object response mode returns a top-level object, so the parser has to accept `{"segments": [...]}`, not just a bare array. I found that with a test before it could find me in production.

And the honest dashboard. The assignment allows mocked metrics, and product-level analytics like DAU are mocked, but every mock row carries an `is_mock` flag and the UI says so on every figure. Latency, cost, and success rate come from real runs. If I put a number in front of a customer, I want to be able to explain where it came from, and that habit should not relax for a take-home.

And the budget cap is a gate, not just a readout. A generation whose projected cost would cross `budget_cap_usd` is refused before the paid ElevenLabs call, and the episode records why it stopped. The cap also counts the OpenAI tokens a failed run already spent, so a run that dies after writing its script still lands against the budget instead of vanishing from it. It was the smallest high-leverage thing to add, and it turns the cost dashboard from advisory into load-bearing: the number on screen is one the system will actually enforce.

And the failure paths are boring on purpose. The outbound OpenAI and ElevenLabs calls retry only on transient errors (timeouts, dropped connections, rate limits, 5xx), reraise the original exception instead of burying it in a wrapper, and run under a bounded per-call timeout, so a bad key fails fast and a hung socket cannot stall a run. The two mutating endpoints validate their bodies, so a missing field or an unknown timezone comes back as a 422 naming the offending field rather than a 500 from deep in a handler, and every error the API returns, validation or domain, speaks one coded envelope. The episode endpoint returns an explicit projection that never includes the on-disk MP3 path. None of this shows in the happy path, which is the point: it is the difference between a demo and something you would leave running.

## Decisions I would revisit with more time

The single-generation lock lives in the database as an episode status. In the first cut, a process that died mid-run left a stuck "generating" row that blocked every future run until someone cleaned it up. I flagged that here, then shipped the fix: `active_generation_exists()` now reclaims any run older than fifteen minutes, marking it failed, before it decides whether a fresh generation may start, so one crash can no longer wedge the endpoint. It is a lazy, single-writer-friendly lease; on Postgres I would push it into the database as a partial unique index on `status = 'generating'` so single-flight is enforced by a constraint rather than a read-then-write.

Dedupe is exact-match on normalized titles and URLs. It catches syndication, but two outlets writing different headlines about the same event both get through, and the prompt is instructed to cut the duplicate. Cheap embedding similarity would do this properly. Episode duration used to sit in this list as an estimate wearing a number's clothes; segmented narration retired that one, since the assembled PCM's byte count is the duration. The estimate survives only in the `NARRATION_MODE=single` fallback.

The frontend test budget is deliberately thin. The load-bearing logic lives in the backend (the pipeline, the cost model, the one-call property, the API contract), so that is where the coverage is concentrated. On the frontend I added two smoke tests over the pieces that carry real display logic, the episode status-chip mapping and the player's speed control, and verified the two pages by hand against the fake backend. A production version would grow that into component tests around the generate-and-poll flow and the dashboard's real-versus-mock rendering.

## How I would extend this for a real customer

This is where the seams pay off. Multiple listeners: preferences become per-user rows and the pipeline already takes preferences as an argument, so fan-out is a loop before it needs to be a queue. When it does need one, the headless CLI boundary is exactly where a durable queue slots in, with cloud cron replacing the user's crontab. The news source protocol swaps RSS for a licensed API the moment there is an SLA conversation. SQLite and local files become Postgres and object storage along a path the code already isolates behind the store layer: the engine reads a `DATABASE_URL`, so moving the database is a connection string plus a psycopg driver and migrations, not a rewrite. I ran it on SQLite because that is the right size for one listener; the URL is the seam that makes Postgres a config change.

Observability is the part I would grow first, because the seed is planted: every stage of every generation already writes a timing-and-outcome event row. For a customer, that becomes alerting on failure rate and latency drift, and the cost dashboard stops being a courtesy and becomes the billing conversation, per-tenant spend against per-tenant caps, with enforcement.

None of that requires rearchitecting, which is the point. The demo is small because the problem is small. The shape is what scales.
